# 📚 SeekBook 二手书与知识资产流转系统

## 📌 项目定位

**SeekBook** 是一个基于 **Vue 3 + Flask** 架构的二手书与知识资产流转系统，设计运行于**完全断网的本地环境**。系统以十万级虚拟书籍数据集为支撑，整合买卖信息发布、知识社区互动，以及基于课程表的知识图谱衍生功能。

### 核心技术架构
- **前端**: Vue 3 + Vite + Composition API，响应式 SPA 设计
- **后端**: Flask + SQLAlchemy + SQLite，RESTful API 架构，微服务蓝图解耦
- **数据层**: 十万级虚拟书籍数据 + SQLite 持久化存储
- **部署环境**: 完全本地化，零外部依赖，断网可运行

## 🏗️ 后端架构设计（200行规范铁律）

### 微服务蓝图工厂模式

系统后端采用 **Flask Blueprint 工厂化架构**，严格遵循 **"任何后端单文件代码量不得超过 200 行"** 的开发铁律：

```
server/
├── app.py              # 应用工厂（仅负责初始化与蓝图挂载）≤ 60行
├── config.py           # 配置中心 ≤ 50行
├── models/             # 数据库模型包
├── init_data.py        # 十万级数据初始化 ≤ 180行
├── run.py              # 启动入口 ≤ 20行
├── views/              # 蓝图视图目录（高内聚低耦合）
│   ├── __init__.py
│   ├── auth.py         # 用户认证蓝图 ≤ 100行
│   ├── books.py        # 书籍管理蓝图 ≤ 150行
│   ├── chats.py        # 即时通讯蓝图 ≤ 120行
│   ├── orders.py       # 订单交易蓝图 ≤ 180行
│   └── isbn_ocr.py     # ISBN智能识别蓝图 ≤ 150行
├── uploads/            # 文件上传持久化目录
└── seekbook.db         # SQLite数据库
```

### 架构设计原则

| 原则 | 说明 |
|------|------|
| **单一职责** | 每个蓝图只处理一类业务逻辑 |
| **接口统一** | 所有 API 返回格式统一为 `{code, message, data}` |
| **数据持久化** | 所有关键数据存入 SQLite，支持断电恢复 |
| **可测试性** | 每个蓝图可独立测试，Mock 数据友好 |

### 1️⃣ 知识星河 —— 社区笔记关联发布与检索

**功能说明**：
- 用户可在**知识星河**页面发布学术笔记，笔记可关联十万级书籍库中的 `book_id`
- 支持**模糊搜索**书名、作者、ISBN，实时下拉匹配关联书籍
- 笔记列表按**时间倒序**排列，新发布笔记**无刷新动态渲染**至顶部
- 每条笔记支持**评论互动**，点击评论按钮弹出详情抽屉，可查看/发布评论

**技术实现**：
- 后端 SQLite 存储 `notes` 和 `comments` 表
- API: `GET /api/community/notes` | `POST /api/community/notes` | `GET /api/notes/{id}/comments` | `POST /api/notes/comment`
- 前端模糊搜索防抖 300ms，限制返回 5 条结果

### 4️⃣ ISBN 智能识别双轨架构（核心亮点）

**功能说明**：
- **第一轨**：对接通义千问 Qwen-VL 视觉大模型，云端 OCR 提取 ISBN
- **第二轨**：文件名作弊后门，纯数字 ISBN 文件名直接查库
- 识别成功后**闪电自动填充**表单（书名、作者、价格、出版社）

**技术实现**：
- 前端 HTML5 拖拽组件，图片 Base64 编码

- 通义千问 API：调用 `qwen-vl-plus` 模型，图片转 Base64 后发送

### 5️⃣ 线下对交交易模式（无余额流转）

**功能说明**：
- 平台**完全无账户余额、无在线充值、无钱包系统**
- 交易流程：聊天协商 → 在线确认见面时间地点 → 线下对交
- 确认后数据**强持久化**至 SQLite（`meet_time`, `meet_place` 字段）
- 刷新/退出重进后，见面信息依然显示

**交易状态流转**：
```
created(已下单) → negotiating(协商中) → confirmed(双方已确认) → completed(线下已完成)
```

**技术实现**：
- 订单表 `orders` 包含 `meet_time`, `meet_place`, `confirmed_at` 字段
- API: `POST /api/orders/{id}/confirm_meet` 确认见面方案
- API: `POST /api/orders/{id}/complete` 标记线下完成
- 前端绿色横幅锁定状态显示

**验证方式**：
1. 进入消息页面 → 选择会话 → 填写见面时间地点
2. 点击"确认线下对交方案" → 绿色横幅显示约定信息
3. 刷新页面（F5）→ 数据依然显示（强持久化验证）

## 🚀 快速启动命令

```bash
# 安装依赖（首次运行）
cd server && pip install -r requirements.txt
cd client && npm install

# 启动服务（两个终端）
# 终端 1 - 后端
cd server && python run.py
# 服务运行在 http://127.0.0.1:5000

# 终端 2 - 前端
cd client && npm run dev
# 服务运行在 http://localhost:5173
```

### 环境变量文件放置位置

- 前端示例文件放在 `client/.env.example`
- 后端示例文件放在 `server/.env.example`
- 真正使用时，把它们复制成同目录下的 `client/.env` 和 `server/.env`

前端变量：
- `VITE_API_BASE_URL`：前端请求 API 的基址，开发环境建议保持为 `/api`
- `VITE_BACKEND_TARGET`：Vite 代理要转发到的后端地址，默认是 `http://127.0.0.1:5000`

后端变量：
- `FLASK_HOST`：后端监听地址
- `FLASK_PORT`：后端监听端口
- `FLASK_DEBUG`：是否开启调试和自动重载，`1` 表示开启
- `CORS_ORIGINS`：允许访问后端的前端地址列表，多个地址用英文逗号分隔

## 📞 技术支持

如遇问题，请检查：
1. 后端服务是否启动（端口 5000）
2. 前端服务是否启动（端口 5173）
3. `server/uploads/` 目录是否有写入权限
4. SQLite 数据库文件 `seekbook.db` 是否存在
